1Problem Overview

A source emits one 160-byte frame every 20 ms — a live audio stream. Frames must cross a hostile UDP relay (drops, jitter, reordering, duplication) and be played out on a fixed schedule at the other end. A frame counts only if it arrives, correct, before its personal deadline: t0 + delay_ms + i·20ms.

Deadline-miss cap
≤ 1.00% of frames may miss their deadline or arrive corrupt
Bandwidth cap
≤ 2.00× the raw stream, counted both directions, drops included
Score
Among valid runs, lowest delay_ms wins

The naive baseline — forward whatever arrives, once, unchanged — fails immediately: even 2% raw packet loss already exceeds the 1% cap at any delay, because a dropped UDP packet is gone forever with no recovery mechanism. This is a loss-recovery problem first, and a delay-minimization problem second.

2System Architecture

Five processes, all UDP, all on 127.0.0.1. The harness (source, relay, player) is fixed and unmodifiable; only the sender and receiver in the middle are ours.

harness source --47010--> SENDER --47001--> relay --47002--> RECEIVER --47020--> harness player
                              ^                                       |
                              |                                       v
                            47004 <----------------- relay <------- 47003
                                    (NACK feedback lane — ARQ backstop)

The harness legs (47010 in, 47020 out) use a fixed wire format we cannot change: 4-byte big-endian sequence number + 160-byte payload. Everything between 47001 and 47004 — the sender↔receiver wire protocol — is our own design, described next.

3Packet Formats

Four message types share the uplink/downlink, distinguished by a leading type byte:

DATA    [0x00][seq  u32 BE][payload 160B]                  = 165 bytes   sender  -> relay -> receiver
PARITY  [0x01][base u32 BE][window u8][xor_payload 160B]   = 166 bytes   sender  -> relay -> receiver
NACK    [0x02][seq  u32 BE]                                =   5 bytes   receiver-> relay -> sender
RETX    [0x03][seq  u32 BE][payload 160B]                  = 165 bytes   sender  -> relay -> receiver

PARITY's base + window together identify exactly which frames it covers: {base, base-1, …, base-window+1}. The receiver strips all of this and re-emits the clean, fixed 4-byte-seq + 160-byte harness format on 47020 — the wire protocol is invisible past the receiver.

4Sender Design

Single-threaded event loop over two sockets (data-in on 47010, NACK-in on 47004), multiplexed with select() so retransmission requests are serviced promptly without blocking on new frames. For every frame received from the harness:

  1. Stash its payload in a retransmission cache (indexed by sequence number) — this is what lets a later NACK be answered without re-fetching anything.
  2. Forward it once, unchanged, as a DATA packet — the harness never sees a delayed or reordered send; the sender cannot send a frame before it has it.
  3. Every 2 frames, XOR the last 3 cached payloads and send the result as a PARITY packet (detailed in §7).

On a NACK, the cached payload is re-sent as a RETX packet — the FEC data path is completely untouched by this addition, which is what keeps the ARQ layer optional and modular (§8).

5Receiver Design

State is indexed by sequence number end-to-end: known/unknown, forwarded-yet, received-as-DATA, reconstructed-via-FEC, delivered-via-RETX, plus per-parity-group bookkeeping. Three responsibilities layer cleanly on top of that state:

  • Duplicate suppression — a relay-duplicated DATA/PARITY/RETX is detected and dropped before it can double-count or re-trigger recovery; each sequence number is forwarded to the player at most once.
  • Reorder tolerance — packets are handled by sequence-number slot, not arrival order, so an early or late arrival for any seq is processed identically; reorder depth is measured, not corrected (no correction is needed for correctness).
  • FEC decode — a fixpoint solver (§7) runs after every DATA or PARITY arrival, resolving whatever it now can.

6Jitter Buffer

The design choice here is easy to get backwards: the receiver never holds a known-good frame to "smooth" playout. The harness player only ever credits the first on-time arrival for a sequence number — so once the receiver has a correct payload (received or FEC-reconstructed), sending it later can only risk the deadline, never help it. The buffer's job is therefore not to add delay but to absorb it: delay_ms is spent entirely as slack for reconstruction to finish before the deadline, not as an artificial release schedule.

Release policy: forward a frame the instant it is known-good. Zero added latency, by design.

7Sliding XOR FEC

Backward-looking sliding-window XOR, rate 1 parity per 2 data frames, window 3:

parity_i = frame_i XOR frame_{i-1} XOR frame_{i-2}          (emitted at even i)

Why backward, not a forward-looking block code

A conventional block FEC groups k future frames and can't emit parity until the group's last member exists — the earliest frame in the group then waits up to (k-1)×20ms for its only chance at repair. Backward-looking parity is built purely from frames the sender already has, so it ships the instant its newest member arrives: no future-wait, no group-completion tax. This alone was found (via closed-form delay modeling, see FEC_DESIGN_REVIEW.md) to cut the achievable minimum delay by roughly 40–50 ms relative to an equivalent-overhead block code.

Fixpoint decoding — chained recovery

The receiver doesn't just solve isolated equations once: any parity with exactly one unknown member is resolved, and because windows overlap (window 3, rate 2), solving one frame can unlock the next equation's only remaining unknown. The solver repeats until nothing more can be recovered in one pass — this is what lets the design repair some consecutive-pair losses that a single non-overlapping block code could not.

8ARQ Evaluation

A retransmission backstop targets exactly the failure FEC cannot fix: two losses inside one 3-frame window. The receiver NACKs a sequence number only when both:

  1. FEC has been given every chance — the frame is still missing after its last covering parity would have arrived, and
  2. a full round trip still fits before the deadline (deadline − now > 2×one-way-delay + slack).
Result in the scored range (40–120 ms): zero NACKs ever fired. The round trip a NACK+resend requires costs roughly 2× the one-way transit time — about 90–290 ms depending on profile — which is larger than any competitive delay budget. Condition (2) above is simply never satisfiable at these delays.

The mechanism itself was confirmed to work at a deliberately generous delay (350 ms, profile B): 6 NACKs fired, 4 frames recovered by retransmission, and the residual unrecoverable count dropped from 3 to 0, at a negligible bandwidth cost (30 feedback bytes). ARQ is correctly implemented and correctly dormant — it is simply the wrong tool for a low-delay-optimized score.

Reordering, duplication, isolated loss, and even delay spikes are all handled robustly (spikes are partly absorbed for free — a spike-delayed frame's independently-routed parity often reconstructs it before the deadline). Burst-correlated loss is the one impairment this design does not survive, and — critically — the failure does not respond to added delay: D and E's miss rates are flat across the entire 50–120ms range, because the misses are unrecoverable residual, not lateness.

10Limitations

  • Burst loss defeats single-parity XOR. Any 2+ consecutive losses inside one 3-frame window are structurally unrecoverable — confirmed by the unrecoverable_groups instrumentation counting stuck parity equations directly.
  • ARQ cannot compensate at competitive delay. Its round-trip cost only fits the budget well above the scored delay range (§8).
  • Fixed FEC parameters. Rate (1:2) and window (3) are static, not adapted to the observed loss pattern of a given run.

11Future Improvements

Interleaving

Build parity from frames spaced by a stride wider than the expected burst length instead of consecutive ones. A burst of D consecutive losses then spreads across D different parity groups — one loss each, already recoverable by the existing solver. No overhead change; targets the exact failure mode found in §9.

Second parity stream

A second, independent parity per window raises per-group recovery capacity from 1 to 2 erasures, fixing the common adjacent-double case directly. Weaker alone against long bursts (3+) than interleaving, but complementary — the two combined would cover both clustered and spread loss patterns.
